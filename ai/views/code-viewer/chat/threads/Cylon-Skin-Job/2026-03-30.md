## 🔍 The Problem

**There are TWO different "session" concepts at play:**

1. **Kimi CLI Session** (`--session <id>`): Persisted conversation context managed by Kimi CLI itself
2. **IDE Thread**: The UI's representation of a conversation

### Root Cause: Kimi CLI Session Restoration

When the server spawns Kimi with `--session <uuid>`, Kimi CLI **restores the previous context** for that session ID from its internal storage. This is by design — it allows resuming conversations.

**But the issue is:**
- When you create a **new thread** via `thread:create`, it generates a fresh UUID ✓
- When you open an **existing thread** via `thread:open` or `thread:open-daily`, it reuses the thread ID
- Kimi CLI restores context based on `--session <threadId>`

### The Daily Thread Issue

The `thread:open-daily` uses today's date as the ID:
```javascript
const todayId = `2026-03-30`;  // Same ID all day
```

So every time you "start fresh" on the same day, you get:
```
kimi --wire --yolo --session 2026-03-30
```

Kimi CLI sees `--session 2026-03-30` and thinks "I know this session!" and restores all previous context.

## 🛠️ Potential Fixes

**Option 1: Don't use `--session` for new threads**
Remove `--session` flag entirely for fresh threads (Kimi starts truly fresh).

**Option 2: Add random suffix to daily threads**
```javascript
const todayId = `2026-03-30-${uuidv4().slice(0,8)}`;
```

**Option 3: Add a "New Fresh Session" button**
A separate action that spawns Kimi without `--session`.

**Option 4: Send `/clear` to Kimi on new thread**
After spawning, send the `/clear` slash command to reset context.
